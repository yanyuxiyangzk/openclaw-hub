# 架构分析：数据库设计

## 一、核心表结构

| 表名 | 用途 | 关键字段 |
|------|------|----------|
| users | 用户认证 | id, email(U), password_hash |
| organizations | 租户 | id, owner_id |
| organization_members | 租户成员 | org_id, user_id, role |
| projects | 项目 | org_id, status, created_by |
| agents | Agent实体 | org_id, agent_type, status |
| project_agents | 项目-Agent关联 | project_id, agent_id (U) |
| project_members | 项目成员 | project_id, user_id, role |

## 二、索引设计

**已有索引：**
- `ix_users_email` - 用户邮箱唯一查询
- `ix_invitations_email` - 邀请查询
- `uq_project_agent` - project_agents联合唯一约束

**缺失索引（建议添加）：**
- `projects(org_id)` - 项目列表查询
- `agents(org_id)` - Agent列表查询
- `project_members(project_id)` - 项目成员查询
- `project_members(user_id)` - 用户参与项目查询

## 三、多数据源适配

**当前实现：**
- SQLite开发环境：`check_same_thread=False`
- PostgreSQL生产环境：生产配置分离

**改进建议：**
1. UUID策略：当前使用TEXT模拟UUID，建议生产环境切换至SQLAlchemy UUIDType
2. 跨数据库兼容：JSON字段在SQLite用TEXT，PostgreSQL用JSONB
3. 连接池配置：生产环境需配置pool_size和max_overflow

## 四、软删除设计

projects表使用`status`字段标记删除（active/archived/deleted），通过ORM relationship的cascade配合实现级联。

## 五、总结

表结构设计合理，外键级联删除配置完善。索引可针对高频查询场景补充。多数据源适配需注意UUID类型和JSON字段的方言差异。
